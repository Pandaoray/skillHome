# AGENTS.md 自动生成任务指令

> **目标读者**：具备代码读取与分析能力的 AI Agent  
> **使用场景**：中大型工程项目 / 多模块系统 / 需要长期演进的代码仓库  
> **核心理念**：以“认知建模优先”，而非“文档堆砌” [优化点：添加版本号以追踪迭代]

---

## 0. 核心目标（Objective）

你的任务是为当前项目构建一套 **完整、可执行、可维护的 `AGENTS.md` 文档体系**，以支持 **Human-assisted AI Development（人辅助 AI）** 模式。

### 目标不是“解释代码”，而是

- 将 **隐性工程知识**（设计意图、默认约束、真实依赖）转化为 **显性结构化文档**
- 使 Agent **无需通读代码**，仅通过 `AGENTS.md` 即可：
  - 建立系统心智模型
  - 快速定位修改切入点
  - 规避高风险副作用

---

## 1. 交付标准（Definition of Done, DoD）

你的输出必须 **同时满足以下所有条件**，否则视为未完成。

### 1.1 覆盖性（Coverage）

- 必须生成 `AGENTS.md` 的目录：
  - 项目根目录
  - 所有 **一级源码目录**（e.g., src/, app/）
  - 所有 **子目录**（递归覆盖所有包含代码或配置的子目录；跳过纯辅助目录，如文件数 < 3 且无业务逻辑/外部依赖）
  - 关键 **配置 / 脚本 / Job / Migration / Infra** 目录（e.g., config/, scripts/）
- 目录存在但未生成 `AGENTS.md` → **视为缺陷** [优化点：扩展为递归全覆盖子目录，实现“每个相关子目录”期望；添加智能跳过以平衡效率]

### 1.2 真实性（Groundedness）

- 文档内容 **必须基于真实代码扫描**
- 必须包含：
  - 真实存在的 **类 / 函数 / 文件名**
  - 实际的 **调用关系或流程**
  - 明确的 **外部资源引用**（如 Redis Key、Kafka Topic、DB 表）
- ❌ 严禁：
  - 编造类名
  - 泛化描述（如“处理相关逻辑”“提供通用能力”）

### 1.3 自包含（Self-Contained）

- 每一份 `AGENTS.md` 都必须：
  - 独立可读
  - 不依赖读者已理解其他模块
- Agent **不应被迫频繁跳转源码** 才能理解模块职责

### 1.4 可操作性（Actionability）

- 文档必须帮助 Agent 回答以下问题：
  - “**我要改这个需求，从哪里下手？**”
  - “**改这里会影响哪些地方？**”

---

## 2. 明确禁止的反模式（Anti-Patterns）

以下任一情况出现，必须重写对应文档：

- ❌ 使用模糊表述而未列出具体类 / 方法
- ❌ 只描述“目录职责”，不描述“代码结构”
- ❌ 假设读者已了解上下游系统
- ❌ 回避副作用（缓存、异步、幂等、补偿、延迟一致性）
- ❌ 文档看起来“正确”，但无法指导实际修改
- ❌ 暴露敏感信息（如API密钥、凭证）

---

## 3. 执行流程（Process）

请**严格按阶段执行**，不得跳步。如果目录为空或非代码相关（e.g., docs/），可跳过生成但需在根目录Map中标注。

---

### Phase 1：全景侦察（Reconnaissance）

#### 目标

构建对项目的 **结构级认知**，而非业务细节。

#### 操作

- 使用目录递归工具（如`tree -L 3`命令或MCP工具）扫描整个仓库
- 输出一份 **目录结构清单（供内部分析，不必写入文档）**
- 标注：
  - 业务模块（e.g., feature-specific）
  - 基础设施层（e.g., utils, infra）
  - 配置 / 脚本 / 自动化相关目录

---

### Phase 2：根目录 `AGENTS.md` 构建

**文件路径**：`{ProjectRoot}/AGENTS.md`

#### 必须包含以下内容

##### 1️⃣ 系统概述（System Overview）

- 项目解决什么问题？
- 核心业务边界是什么？
- 主要技术栈（语言 / 框架 / 存储 / 消息系统）

##### 2️⃣ 目录导航（Directory Map）

| 目录 | 职责 | 关键说明 | AGENTS.md 链接 |
|---|---|---|---|
| `xxx/` | | | [链接或路径] |

##### 3️⃣ 核心业务场景索引（Critical）

列出 **5–20 个真实存在的核心业务场景**（优先选择高频调用、关键路径或业务 criticality，根据项目规模调整数量），并给出 **明确代码切入点**：

- **场景名称**（e.g., 用户注册）
  - 入口：`Controller / Handler` (e.g., UserController.register)
  - 核心逻辑：`Service / Domain` (e.g., UserService.createUser)
  - 异步 / 副作用：`Producer / Consumer / Job` (e.g., EmailProducer.sendWelcome) [优化点：增加数量到5-10，并明确选择标准，适合中大型项目]

##### 4️⃣ 全局设计约束（Global Constraints）

- 事务 / 一致性约束
- 缓存使用规则
- 异步消息使用原则
- 明确的 **“禁止事项”** (e.g., 禁止直接SQL查询)

---

### Phase 3：子目录级 `AGENTS.md` 构建（迭代执行）

对每一个目录及其子目录（递归），执行以下循环：

#### Step 1：Scanning

- 列出目录下所有文件与子目录（忽略.git、node_modules等）

#### Step 2：Analysis

- 阅读 **所有代码文件**
- 识别：
  - 核心类（e.g., >3次调用的类）
  - 关键流程
  - 外部依赖

#### Step 3：Writing

- 在该目录下生成 `AGENTS.md` [优化点：明确递归，确保子子目录覆盖]

---

## 4. 文档模板

### 模板 A：业务模块（business / domain / feature）

```markdown
# {ModuleName} 模块开发指南

> 最后更新：{自动填充当前日期}
> 位置：`{RelativePath}`

## 1. 模块概述
明确说明该模块**解决的业务问题**，而非技术职责。（示例：处理用户认证，而不是“提供登录功能”）

## 2. 核心代码结构
| 文件/目录 | 职责 | 关键类 / 方法 |
|---|---|---|

## 3. 核心业务流程
- **场景 A**：请求 → 校验 → 核心处理 → 副作用（用箭头表示流程，标注类名）
- **场景 B**：…

## 4. 关键资源与副作用
- Redis：
  - Key：`xxx`
  - 结构 / TTL / 写入时机
- MQ / Kafka：
  - Topic：`xxx`
  - 生产 / 消费语义
- DB：
  - 关键表与字段

## 5. 常见修改场景与切入点
- 新增规则：修改 `XXXService#method`
- 性能问题：优先检查 `xxx` 缓存
- 行为异常：检查异步消费是否幂等

## 6. 维护与风险说明
- 修改该模块可能影响的下游模块（列出2-3个）
- 高风险点与回归建议（使用风险矩阵：| 风险 | 概率 | 影响 | 缓解 |）
```

### 模板 B：通用 / 基础设施层（utils / common / config / infra）

```markdown
# {LayerName} 层开发指南

> 最后更新：{自动填充当前日期}
> 位置：`{RelativePath}`

## 1. 概述
该层在系统中的**定位与边界**。（示例：提供跨模块的缓存抽象，不处理业务逻辑）

## 2. 核心组件
- `ClassA`：用途 + 使用约束（示例：必须在事务外调用）
- `ClassB`：用途 + 常见误用

## 3. 设计约定
- 状态性 / 无状态要求
- 线程安全 / 并发假设
- 禁止直接依赖的模块

## 4. 使用示例或典型调用路径
简要说明典型调用方式，帮助快速理解。（添加代码片段或伪代码）
```

---

## 5. 质量校验（Self-Correction Checklist）

在提交前，逐条自检：

- [ ] 是否存在未列出具体类名的模块说明？（至少5个具体引用）
- [ ] 是否有相关目录未生成 `AGENTS.md`？（检查递归覆盖）
- [ ] 根目录索引是否能准确跳转到子模块？
- [ ] 是否明确标注了副作用与风险点？（至少2个量化风险）
- [ ] 文档是否真的能指导一次“真实修改”？（模拟一个修改场景验证）
- [ ] 是否考虑了多语言/混合仓库？（如果适用，标注）

如任一项不满足 → **返回对应 Phase 重做**。

---

## 6. 输出要求（Final Output Rules）

- 使用 **Markdown**
- 不输出中间分析过程（可选：如果调试模式，输出日志）
- 不省略任何相关目录
- 以 **工程师交接文档** 的严谨度为标准
- 可选扩展：如果工具支持，添加流程图（e.g., Mermaid）以可视化核心流程

> **记住**：
> `AGENTS.md` 的质量，决定了 Agent 是否能成为“合格的协作者”，而不是“制造噪音的工具”。
