# Claude Code Skill: myGod (大神/大仙) v1.0

## 一、概述

设计一个 Claude Code Skill，用于中大型项目的「人辅助 AI 编程」工作流。通过 Agent Team 模式，将 PM 的简单需求自动转化为完整的开发流程，实现从需求澄清到功能验证的全链路自动化。

### 核心理念

- **人辅助 AI**：人只在关键决策点介入（需求澄清、最终确认）
- **全程可追溯**：每个阶段产出文档，决策过程透明
- **渐进式验证**：尽早发现问题，降低修复成本
- **智能降级**：异常时优雅降级，而非直接失败

## 三、执行模式

### 1. 完整模式 (Full Mode) - 默认

```bash
/myGod "实现用户登录功能"
```

- 完整 7 步流程
- 适用于新功能、复杂需求
- 预计耗时：30-60 分钟

### 2. 快速模式 (Quick Mode)

```bash
/myGod quick "给用户表添加手机号字段"
```

- 跳过架构设计
- 简化验证（仅运行相关测试）
- 适用于小功能、明确需求
- 预计耗时：10-20 分钟

### 3. 热修模式 (Hotfix Mode)

```bash
/myGod hotfix "修复登录页面500错误"
```

- 最小流程：澄清 → 开发 → 基础验证
- 跳过架构、拆解、审查
- 适用于紧急 bug 修复
- 预计耗时：5-15 分钟

### 模式对比

| 阶段 | 完整模式 | 快速模式 | 热修模式 |
|------|:--------:|:--------:|:--------:|
| 需求澄清 | ✅ 完整 | ✅ 简化(1轮) | ✅ 简化(1轮) |
| 架构设计 | ✅ | ❌ 跳过 | ❌ 跳过 |
| 需求拆解 | ✅ | ✅ 简化 | ❌ 跳过 |
| 开发实现 | ✅ 并行 | ✅ 单任务 | ✅ 单任务 |
| 单元验证 | ✅ | ✅ | ❌ 跳过 |
| 集成协调 | ✅ | ❌ 跳过 | ❌ 跳过 |
| 代码审查 | ✅ | ✅ 简化 | ❌ 跳过 |
| 功能验证 | ✅ 完整 | ✅ 相关测试 | ✅ 冒烟测试 |
| Git 集成 | ✅ | ✅ | ✅ |

## 四、Agent Team 角色定义

### 1. 🎯 需求澄清师 (Requirement Clarifier)

**职责**：接收 PM 初始需求，通过结构化问答明确具体需求

**输入**：

- PM 的原始需求描述

**输出**：

- `01-clarification.md` 需求澄清文档
- 结构化的验收标准清单

**行为规范**：

- 最多进行 4 轮问答
- 每轮提出 3-8 个针对性问题
- 问题类型：单选、多选、文本输入
- 自动识别需求中的模糊点和缺失信息
- 生成 SMART 验收标准（具体、可衡量、可实现、相关、有时限）

**人工交互**：✅ 需要人工回答澄清问题

---

### 2. 🏗️ 架构师 (Architect)

**职责**：分析项目架构，设计技术方案，评估风险

**输入**：

- 澄清后的需求文档
- 项目现有架构文档（agents.md、readme.md、目录结构）

**输出**：

- `02-architecture.md` 技术方案文档
- `decisions/ADR-xxx.md` 关键决策记录（可选）

**行为规范**：

- 分析现有项目结构和技术栈
- 确定技术选型和实现方案
- 识别潜在风险并给出缓解措施
- 定义模块边界和接口契约
- 若 agents.md 不存在，自动分析项目生成临时角色定义

**输出文档结构**：

```markdown
# 技术方案

## 1. 现状分析
- 项目技术栈
- 相关模块分析
- 现有代码评估

## 2. 方案设计
- 整体架构
- 模块划分
- 接口定义

## 3. 技术选型
- 选型决策及理由

## 4. 风险评估
| 风险 | 等级 | 缓解措施 |
|------|------|---------|

## 5. 工作量预估
```

**人工交互**：❌ 自动执行

---

### 3. 📋 需求拆解师 (Task Decomposer)

**职责**：将需求拆解为可执行的开发任务，分析任务依赖

**输入**：

- 澄清后的需求
- 技术方案文档

**输出**：

- `03-task-decomposition.md` 任务拆解文档

**行为规范**：

- 按模块/功能/层级拆解任务
- 每个任务粒度控制在 15-30 分钟可完成
- 明确任务依赖关系，生成依赖图
- 标注可并行执行的任务
- 为每个任务定义清晰的输入/输出

**输出文档结构**：

```markdown
# 任务拆解清单

## 依赖关系图
task-01 ──→ task-02 ──→ task-04
              ↑
task-03 ──────┘

## 任务列表

### Task-01: [任务名称]
- **描述**: xxx
- **依赖**: 无
- **输入**: xxx
- **输出**: xxx
- **验收标准**: xxx
- **预估时间**: 15min
- **可并行**: 是

### Task-02: [任务名称]
...
```

**人工交互**：❌ 自动执行

### 4. 💻 开发者 (Developer) - 动态多实例

**职责**：根据分配的任务进行具体编码实现

**输入**：

- 具体任务描述
- 上下文摘要（压缩后的关键信息）
- 相关代码文件

**输出**：

- 代码实现
- `04-development/task-xx-xxx.md` 实现说明文档

**行为规范**：

- 根据任务拆解结果动态创建实例
- 按依赖关系调度，最大化并行执行
- 遵循项目现有代码风格
- 完成后立即触发单元验证
- 记录实现决策和注意事项

**输出文档结构**：

```markdown
# Task-01 实现说明

## 变更文件
| 文件 | 操作 | 说明 |
|------|------|------|
| src/auth/login.ts | 新增 | 登录逻辑 |

## 实现要点
- xxx

## 注意事项
- xxx

## 单元验证结果
- ✅ 测试通过: 5/5
```

**人工交互**：❌ 自动执行

### 5. 🧪 单元验证器 (Unit Validator) - 新增

**职责**：对单个开发任务进行即时验证

**输入**：

- 开发者的代码变更
- 任务的验收标准

**输出**：

- 单元验证结果（嵌入到开发者文档中）

**行为规范**：

- 每个开发者完成后立即执行
- 运行相关单元测试
- 检查代码基本规范（lint）
- 验证是否满足任务验收标准
- 失败则立即反馈给对应开发者修复

**人工交互**：❌ 自动执行

---

### 6. 🔗 集成协调者 (Integrator)

**职责**：合并多个开发者的代码，解决冲突

**输入**：

- 所有开发者的代码变更
- 各任务的依赖关系

**输出**：

- `05-integration.md` 集成报告
- 集成后的代码（合并到目标分支）

**行为规范**：

- 所有开发者完成后统一执行
- 按依赖顺序合并代码
- 自动解决简单冲突
- 复杂冲突记录并标记
- 运行集成测试验证

**人工交互**：❌ 自动执行

---

### 7. 🔍 代码审查员 (Code Reviewer)

**职责**：审查代码质量，确保符合项目规范

**输入**：

- 集成后的完整代码变更

**输出**：

- `06-code-review.md` 代码审查报告

**行为规范**：

- 检查代码风格一致性
- 检查潜在 bug 和安全问题
- 检查性能问题
- 检查测试覆盖率
- 问题按严重程度分级（阻塞/警告/建议）
- 阻塞级问题必须修复后才能继续

**输出文档结构**：

```markdown
## 总体评价
- 通过 / 需修改

## 问题列表

### 🔴 阻塞级 (必须修复)
| 文件 | 行号 | 问题 | 建议 |
|------|------|------|------|

### 🟡 警告级 (建议修复)
...

### 🟢 建议级 (可选优化)
...

## 亮点
- xxx
```

**人工交互**：❌ 自动执行

---

### 8. ✅ 功能验证师 (Validator)

**职责**：验证实现是否满足需求

**输入**：

- 澄清后的需求（验收标准）
- 最终代码

**输出**：

- `07-validation.md` 功能验证报告
- `08-final-checklist.md` 最终验证清单

**验证方式**：

1. **自动测试**：运行现有测试套件
2. **需求对照**：逐条验证验收标准
3. **生成测试**：根据需求自动生成并运行测试
4. **人工清单**：生成验证清单供人工确认

**行为规范**：

- 验证失败最多重试 3 次
- 每次失败生成详细原因分析
- 超过重试次数后请求人工介入
- 生成可供人工执行的验证清单

**人工交互**：✅ 最终验证结果需人工确认

## 六、上下文压缩机制

### 设计目标

- 避免 token 爆炸导致 AI 响应质量下降
- 确保每个角色获得必要且充分的上下文

### 摘要格式标准

每个阶段完成后，生成结构化摘要供下游角色使用：

```markdown
## Context Summary

### 🔴 Must Know (必须知道)
关键决策和硬性约束，下游角色必须遵守
- 使用 JWT 进行身份认证（非 Session）
- API 路径前缀: /api/v2/
- 数据库使用现有的 PostgreSQL

### 🟡 Should Know (应该知道)
重要背景信息，有助于做出更好决策
- 现有用户表结构: users(id, email, password_hash, created_at)
- 项目使用 TypeScript strict 模式

### 🟢 Reference (参考文档)
详细信息的文档链接，需要时查阅
- 完整需求: ./01-clarification.md
- 技术方案: ./02-architecture.md#authentication
- 相关代码: src/middleware/auth.ts
```

### 摘要传递规则

| 当前角色 | 接收摘要来源 |
|---------|-------------|
| 架构师 | 需求澄清师 |
| 需求拆解师 | 需求澄清师 + 架构师 |
| 开发者 | 需求拆解师（仅相关任务） |
| 集成协调者 | 需求拆解师（依赖关系） |
| 代码审查员 | 架构师（规范约束） |
| 功能验证师 | 需求澄清师（验收标准） |

## 八、Git 集成

### 自动化行为

1. **分支创建**：工作流开始时自动创建 feature 分支

   ```
   feature/myGod-{简短描述}-{timestamp}
   ```

2. **提交策略**：
   - 每个开发任务完成并通过单元验证后，自动 commit
   - Commit message 格式：`[myGod] task-xx: 简要描述`

3. **PR 创建**：功能验证通过后自动创建 PR
   - 标题：`[myGod] 需求简述`
   - 描述：自动生成，包含需求摘要、变更列表、验证结果


## 九、超时与降级机制

### 超时设置

| 角色 | 默认超时 | 超时处理 |
|------|---------|---------|
| 需求澄清师 | 30分钟/轮 | 提醒用户继续或使用当前信息推进 |
| 架构师 | 10分钟 | 使用简化方案（跳过风险评估） |
| 需求拆解师 | 10分钟 | 粗粒度拆解 |
| 开发者 | 20分钟/任务 | 标记任务为复杂，建议人工介入 |
| 单元验证器 | 5分钟 | 跳过自动测试，标记待验证 |
| 集成协调者 | 15分钟 | 顺序合并避免复杂冲突 |
| 代码审查员 | 10分钟 | 降级为基础 lint 检查 |
| 功能验证师 | 15分钟 | 生成部分报告，标记未验证项 |

### 降级策略

```
全功能 → 简化功能 → 最小功能 → 人工接管

示例：
- 自动生成测试失败 → 仅运行现有测试 → 仅 lint 检查 → 人工验证
- 代码审查超时 → 基础规范检查 → 跳过审查(标记) → 人工审查
- 集成冲突复杂 → 顺序执行避免冲突 → 人工解决冲突
```

## 十四、技术实现要点

1. **Skill 入口**: 创建 `.claude/skills/myGod.md` 定义 Skill
2. **Agent 调度**: 使用 Claude Code 的 Task 工具启动子 Agent
3. **并行控制**: 开发者角色使用 `run_in_background` 参数并行执行
4. **状态管理**: 通过 `workflow-state.json` 持久化工作流状态
5. **上下文传递**: 通过 `context-summaries/` 目录实现角色间信息共享
6. **Git 操作**: 使用 Bash 工具执行 git 命令和 gh CLI
